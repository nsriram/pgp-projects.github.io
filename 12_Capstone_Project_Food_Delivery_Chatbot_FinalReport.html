1. Executive Summary

This report documents the complete development of an AI-powered customer service chatbot for FoodHub, a food delivery aggregator. The chatbot autonomously handles common customer order queries, reducing dependency on manual support teams.

The project is structured in six technical stages:

  1. LLM Setup - GPT-5-mini, a reasoning-capable model, is configured with reproducible parameters and connected to an MLFlow observability layer for full tracing of all model and tool interactions.

  2. Question Answering LLM - The model is evaluated on nine benchmark customer queries under two conditions: a generic system prompt (baseline) and a FoodHub-specific refined prompt. The refined prompt lifts average accuracy from 2.22/5 to 4.22/5 (+90%) and clarity from 2.78/5 to 4.44/5 (+60%).

  3. SQL Agent - A LangChain SQL Agent is built on top of the SQLite order database. It is equipped with four tools (table listing, schema inspection, query validation, query execution) and verified against direct database queries, achieving exact matches on all 10 columns for every tested order.

  4. Chat Agent - Two LangChain tools are designed: order_query_tool (wraps the SQL Agent) and answer_tool (composes polished customer responses). These tools are assembled into a FoodHub Chat Agent with layered guardrails - input guardrails in the Chat Agent system prompt and output guardrails in the Answer Tool prompt.

  5. Chatbot Testing - Seven test queries are run end-to-end. All four problem-statement questions and three additional pipeline-verification queries produce correct responses. The chatbot achieves 7/7 accuracy across security refusal, escalation, Order ID gating, and live database queries.

  6. Deployment - The chatbot is packaged as a Dockerised Streamlit application and deployed to a public Hugging Face Space via a fully automated GitHub Actions CI/CD pipeline. The live application is available at https://huggingface.co/spaces/nsriram78/FoodHubChatbot.

2. Business Context and Objective

2.1 Business Context

The number of online food delivery orders is increasing rapidly in cities, driven by students, working professionals, and families with busy schedules. Customers frequently raise queries about their orders - delivery time, order status, payment details - and currently these queries are managed manually by customer support teams. This results in long wait times, inconsistent responses, and higher operational costs.

FoodHub wants to enhance customer experience by introducing automation. Since the app already maintains structured order information in its database, there is a strong opportunity to leverage this data through intelligent systems that interact directly with customers in real time.

2.2 Objective

Design and implement a functional AI-powered chatbot that: - Connects to the order database using a SQL Agent to fetch accurate order details - Converts database results into concise, polite, customer-friendly responses - Applies input and output guardrails to ensure safe interactions and prevent misuse - Escalates queries to human agents when necessary

2.3 Dataset Description

The dataset is sourced from the company’s order management database (SQLite). It contains a single orders table with the following columns:

Column Description
order_id Unique identifier for each order
cust_id Customer identifier
order_time Timestamp when the order was placed (HH:MM)
order_status Current status (e.g., placed, preparing food, out for delivery, delivered)
payment_status Payment confirmation details
item_in_order Comma-separated list of items in the order
preparing_eta Estimated preparation completion time (HH:MM)
prepared_time Actual preparation completion time (HH:MM)
delivery_eta Estimated delivery time (HH:MM)
delivery_time Actual delivery time (HH:MM); NULL if not yet delivered

Time columns are stored as TEXT in HH:MM format. NULL values indicate a stage has not been reached yet.

6. Build SQL Agent

6.1 Database Setup

The SQLite database is loaded using SQLDatabase.from_uri(). This creates a LangChain-compatible database object that the SQL Agent can inspect and query.

Database Schema:

CREATE TABLE orders (
    order_id       TEXT,
    cust_id        TEXT,
    order_time     TEXT,
    order_status   TEXT,
    payment_status TEXT,
    item_in_order  TEXT,
    preparing_eta  TEXT,
    prepared_time  TEXT,
    delivery_eta   TEXT,
    delivery_time  TEXT
)

All time columns are stored as TEXT in HH:MM format. The item_in_order column is a comma-separated string.

6.2 SQL Agent Design

6.2.1 System Prompt

The SQL Agent system prompt addresses the core gap from §5 - LLM responses were domain-aware but not data-grounded. Key design decisions:

Design Decision Rationale
Inject db.dialect into prompt Generates dialect-correct SQL (SQLite vs PostgreSQL)
Explicit time arithmetic formula Prevents hallucinated date functions for HH:MM text columns
Explicit item count formula Prevents incorrect COUNT on comma-separated text
NULL handling instruction Prevents incorrect interpretation of in-progress orders
JSON output format Structured output that answer_tool can reliably parse
DML prohibition Prevents INSERT/UPDATE/DELETE by the agent

6.2.2 SQL Agent Tools

The SQL Agent is equipped with four tools from SQLDatabaseToolkit:

Tool Role
sql_db_list_tables Lists all tables in the database
sql_db_schema Inspects the schema of a specific table
sql_db_query_checker Validates SQL before execution
sql_db_query Executes the validated SQL query

These four tools implement a reliable query pipeline: inspect schema → write query → validate → execute.

7. Build Chat Agent

7.1 Order Query Tool

7.1.1 Design

The Order Query Tool is the first stage in the chat pipeline. It bridges the customer’s natural language input and the SQL Agent backend built in §6.

Tool Name: order_query_tool The name is intentionally self-descriptive - the Chat Agent’s LLM uses this name to decide when to invoke the tool.

Parameter: query - the customer’s natural language question, including the Order ID when known. The parameter description provides examples so the Chat Agent frames the query correctly.

Return value: A JSON string with order data, or {"error": "Order not found"} if the order does not exist.

Implementation: The tool is a thin wrapper around run_sql_agent_query() - it passes the query to the SQL Agent and returns the raw result. No transformation is done here; transformation is the responsibility of answer_tool.

7.1.2 Tool Schema (as seen by the Chat Agent LLM)

Tool Name        : order_query_tool
Tool Description : Query the FoodHub order database for order-related information.
                   Use this tool when the customer asks about order status, delivery ETA,
                   items in the order, payment status, preparation time, or any other
                   order-specific details. Include the Order ID (e.g. 'O12486') in the
                   query when known for a precise database lookup.

7.2 Answer Tool

7.2.1 Design

The Answer Tool is the second stage in the chat pipeline. It takes the raw database output from the Order Query Tool and the customer’s original question, and composes a polished, customer-facing response.

Tool Name: answer_tool Parameters: - customer_query - the customer’s original question, exactly as submitted - raw_order_data - the complete JSON response from order_query_tool

Output guardrails are embedded in the Answer Tool’s system prompt (ANSWER_TOOL_PROMPT):

Guardrail Description
Strict Focus Rule Answer only what was asked; no unsolicited fields or summaries
No proactive offers Do not ask follow-up questions or suggest actions unprompted
Accuracy constraint Only reference data explicitly present in the raw JSON
NULL handling If delivery_time is null, state order not yet delivered - do not guess
No policy promises Do not state refund timelines unless present in the data
Escalation phrase Use exact phrase when escalation is indicated in the raw data

Implementation: The tool invokes model_gpt5_mini directly with the output guardrail prompt and a human message containing both the customer query and the raw data. This separates output-side control from the Chat Agent’s input-side control.

7.3 Combined Tools

In LangChain, tools do not call each other directly. “Combining” means assembling both tools into a single list registered with the Chat Agent. The orchestration - which tool to call first, what to pass between them - is handled by the Chat Agent at inference time, guided by the system prompt.

chat_tools = [order_query_tool, answer_tool]

Tools registered: - order_query_tool - database retrieval - answer_tool - response composition

7.4 FoodHub Chat Agent

7.4.1 Design

The Chat Agent is the top-level orchestrator. It receives the raw customer message, applies input guardrails, and coordinates the two tools.

Input guardrails are embedded in the Chat Agent system prompt (CHAT_AGENT_SYSTEM_PROMPT):

Guardrail Description
Security and scope Refuse hacking requests and out-of-scope queries without calling any tool
Order ID requirement Ask for Order ID before calling order_query_tool if not provided
Tool sequencing Always call order_query_tool first, then answer_tool
Escalation Detect repeated complaints and escalate immediately

Why two separate prompt locations?

Input and output guardrails are intentionally placed in separate prompts: - CHAT_AGENT_SYSTEM_PROMPT governs what the Chat Agent decides to do (input-side) - ANSWER_TOOL_PROMPT governs how the response is composed (output-side)

Separating them keeps each layer focused. The Chat Agent does not need to know how to write customer responses; the Answer Tool does not need to know about security rules.

7.4.2 Agent Verification

After assembly, the Chat Agent is verified:

Chat Agent type  : CompiledStateGraph
Tools registered : ['order_query_tool', 'answer_tool']
System prompt    : CHAT_AGENT_SYSTEM_PROMPT (input guardrails + tool sequencing)
Answer prompt    : ANSWER_TOOL_PROMPT (output guardrails)

The CompiledStateGraph type confirms the agent is a LangGraph compiled graph, which supports streaming and configurable recursion limits.

8. Build a Chatbot and Answer User Queries

8.1 Chatbot Loop Design

The chatbot loop is implemented as two components:

run_chat_agent_query(query) - a reusable helper that takes a single customer query, passes it to the Chat Agent, streams the response, and returns the final answer as a plain string. Handles reasoning model output where content is returned as a list of typed blocks.

chatagent() - the interactive loop that manages the customer-facing session: greeting, input collection, response display, and clean exit. Each customer message is passed to run_chat_agent_query() independently (stateless per query).

LangGraph recursion limit: The nested agent architecture (Chat Agent → order_query_tool → SQL Agent) requires a recursion limit above the default of 25. The SQL Agent alone uses 4 tool call steps. Setting recursion_limit=50 in the stream configuration provides sufficient budget for the full pipeline.

8.3 Observations on Agent Workflow and Accuracy

8.3.1 Two Execution Paths

The Chat Agent follows two distinct execution paths depending on the query:

Path 1 - Guardrail Intercept (no tool calls)

For queries that trigger input guardrails, the Chat Agent responds directly without invoking any tool. This is the correct behaviour - it avoids unnecessary database access and cost.

  • Security queries (Q1): refused immediately, no tools called
  • Escalation queries (Q2): escalated to human agent, no tools called
  • Missing Order ID (Q3, Q4): asks for Order ID, no tools called

Path 2 - Full Tool Pipeline

For valid queries with an Order ID, the Chat Agent follows the two-step pipeline:

  1. order_query_tool → SQL Agent → SQLite DB → raw JSON
  2. answer_tool → LLM + output guardrails → customer-facing response

Pipeline Configuration Note: The LangGraph recursion_limit is set to 50. The SQL Agent’s 4 tool steps (list_tables → schema → query_checker → query) are counted against the shared recursion budget of the outer Chat Agent. The default limit of 25 is insufficient for this nested architecture.

9. Actionable Insights and Recommendations

9.1 Key Takeaways for Business

1. Customer query resolution is now instant for common scenarios

The chatbot handles order status, delivery ETA, item queries, and payment questions in real time - without human intervention. Customers no longer wait in a queue for answers that can be fetched directly from the database.

2. Human agents are protected for high-value interactions

Escalation guardrails automatically route frustrated customers (repeated complaints, unresolved issues) to human agents. This means human attention is reserved for situations where it genuinely adds value, rather than being consumed by routine queries.

3. Security is enforced at the first line of contact

The input guardrail layer intercepts security threats before any database access occurs. This protects customer data and prevents prompt injection attacks from reaching the SQL Agent.

4. The chatbot is honest about its limitations

When an order is not found or data is missing, the chatbot responds with a polite, accurate message rather than guessing or inventing information. This builds customer trust and prevents downstream confusion.

9.2 Key Takeaways for the Technical Team

1. SQL Agent is essential for data-grounded responses

Direct LLM responses for order-specific questions are generic and unverifiable - as demonstrated in §5. The SQL Agent returns exact, database-grounded values. This distinction is critical for accuracy and customer trust.

2. Separate input and output guardrails

Placing input guardrails in the Chat Agent system prompt and output guardrails in the Answer Tool prompt keeps each layer focused. The Chat Agent controls routing decisions; the Answer Tool controls response quality. Mixing both concerns in one prompt reduces maintainability and makes guardrail debugging harder.

3. Monitor and tune the recursion limit

The nested agent architecture (Chat Agent → SQL Agent) consumes LangGraph’s recursion budget faster than a flat agent. The default limit of 25 is insufficient; 50 is appropriate for this design. Any addition of new nested tools should be accompanied by a budget review.

4. Reasoning models improve reliability

GPT-5-mini’s chain-of-thought reasoning provides an implicit check before tool calls - the model reasons about whether a query requires a database lookup before calling order_query_tool. This reduces spurious tool calls on guardrail queries.

9.3 Roadmap

The following improvements are recommended for a production-ready deployment:

1. Scale to production data volumes

The current dataset contains 20 rows - sufficient for development. In production, the orders table would contain millions of records. SQL queries must include indexed lookups on order_id and cust_id. Query timeouts should be enforced at the SQL Agent level.

2. Multi-turn conversation memory

The current chatbot is stateless - each query is independent. Production chatbots need conversation memory so a follow-up like “what about the payment?” correctly refers to the previous order. LangGraph’s checkpointing mechanism supports this without architectural changes.

3. Customer identity verification

The current design fetches order data for any claimed Order ID. A production system should verify the customer owns the order (e.g., match cust_id to a session token) before returning sensitive order details.

4. Extend to order actions

The chatbot currently reads data only. Extending it to support write operations (order cancellation, delivery reschedule) would require adding new tools with appropriate confirmation steps and audit logging - achievable within the same LangChain tool framework.

5. Feedback loop for accuracy improvement

Customer satisfaction ratings and escalation rates can be fed back into prompt refinement cycles. The MLFlow experiment tracking infrastructure built in this project is already positioned to support this systematically.

Appendix C: SQL Agent Tool Call Sequence

For every order query, the SQL Agent follows this deterministic four-step sequence:

1. sql_db_list_tables    → discovers available tables
2. sql_db_schema         → inspects column names and types
3. sql_db_query_checker  → validates the generated SQL
4. sql_db_query          → executes the validated query

This sequence ensures the agent never executes a query without first understanding the schema, and never executes a query that fails validation.

Appendix D: System Prompts

D.1 Chat Agent System Prompt - Input Guardrails (CHAT_AGENT_SYSTEM_PROMPT)

Defines the top-level orchestration and input-side safety rules for the Chat Agent: - Tool usage sequence: always call order_query_tool first, then answer_tool - Security refusal: reject hacking requests and bulk data access without calling any tool - Order ID requirement: ask for Order ID before any DB lookup - Escalation triggers and exact escalation phrase - Scope boundary: FoodHub order queries only

D.2 SQL Agent System Prompt - Data Retrieval Rules (SQL_AGENT_SYSTEM_PROMPT)

Defines how the SQL Agent generates and executes queries: - Inject db.dialect for dialect-correct SQL - Explicit julianday() formula for HH:MM time arithmetic - Item count formula for comma-separated item_in_order column - NULL handling: columns with no data are returned as null in JSON - DML prohibition: no INSERT/UPDATE/DELETE/DROP - Output format: valid JSON with column names as keys

D.3 Answer Tool Prompt - Output Guardrails (ANSWER_TOOL_PROMPT)

Defines how raw database output is converted into a customer-facing reply: - Strict focus rule: answer only what was asked; no unsolicited fields or summaries - No proactive offers or follow-up questions - Accuracy constraint: only reference data explicitly present in the raw JSON - NULL handling: state order not yet delivered - do not guess ETA - No policy promises (e.g. refund timelines) unless present in the data - Escalation phrase trigger when raw data indicates a repeated complaint

Appendix E: Deployment

The FoodHub chatbot is packaged as a Dockerised Streamlit application and deployed to a public Hugging Face Space (nsriram78/FoodHubChatbot) via a fully automated GitHub Actions CI/CD pipeline.

E.1 Application File Structure

foodhub_project/
├── agent.py            ← Backend: LLM, SQL Agent, tools, Chat Agent
├── app.py              ← Frontend: Streamlit chat UI
├── Dockerfile          ← Container definition (Python 3.12, port 7860)
├── requirements.txt    ← Python dependencies
├── customer_orders.db  ← SQLite order database
└── .github/
    └── workflows/
        └── deploy.yml  ← GitHub Actions: auto-deploy on push to main

E.2 Streamlit Frontend

The Streamlit app (app.py) initialises the Chat Agent using @st.cache_resource, which runs initialisation exactly once per container lifetime. All user sessions reuse the same agent instance. The UI provides a chat input field, message history, a loading spinner during query processing, and an error display if agent initialisation fails.

E.3 Dockerfile

The container is built from python:3.12. A non-root user (uid 1000) is created for security. The application runs on port 7860, which is the default port expected by Hugging Face Docker Spaces.

E.4 CI/CD Pipeline - GitHub Actions

Deployment is automated via .github/workflows/deploy.yml. On every push to the main branch, the workflow uploads the project folder to the Hugging Face Space using huggingface_hub. The HF_TOKEN secret is stored in GitHub repository Settings → Secrets and variables → Actions.

E.5 Hugging Face Space

The Hugging Face Space is configured as a Docker-type Space. On push, Hugging Face pulls the uploaded files and rebuilds the container automatically.
